# Build Troubleshooting

Common build failures for GemHub Lite (Expo + React Native, bare Android/iOS folders) and their root causes.

## 1. Android: `Cannot run program "node": error=2, No such file or directory`

### Symptom
Gradle sync or `./gradlew` fails during settings evaluation with:

```
A problem occurred evaluating settings 'android'.
> A problem occurred starting process 'command 'node''
Caused by: java.io.IOException: Cannot run program "node" (in directory ".../android"): error=2
```

### Root Cause
`android/settings.gradle` shells out to Node to resolve native module locations:

```groovy
providers.exec {
  commandLine("node", "--print", "require.resolve('@react-native/gradle-plugin/package.json', ...)")
}
```

When **Android Studio is launched from Finder, Dock, or Spotlight** on macOS, the
GUI-spawned Gradle daemon inherits a stripped PATH (`/usr/bin:/bin:/usr/sbin:/sbin`).
It does **not** see:

- `~/.nvm/versions/node/<version>/bin` (nvm-managed Node)
- `/opt/homebrew/bin` (Apple Silicon Homebrew)
- `/usr/local/bin` (Intel Homebrew — though this *is* in the default GUI PATH on most macOS versions, the binary isn't there by default with nvm/Homebrew installs)

Terminal builds work because zsh loads your full PATH; IDE builds don't.

### Fix (Recommended — One-Time, Permanent)

Symlink `node`/`npm`/`npx` into `/usr/local/bin`, which **is** part of the macOS GUI PATH:

```bash
sudo ln -sf "$(which node)" /usr/local/bin/node
sudo ln -sf "$(which npm)"  /usr/local/bin/npm
sudo ln -sf "$(which npx)"  /usr/local/bin/npx
```

Then in Android Studio: **File → Invalidate Caches → Invalidate and Restart** (clears the Gradle daemon's cached environment).

Verify:

```bash
env -i PATH=/usr/bin:/bin:/usr/local/bin which node
# should print: /usr/local/bin/node
```

### Alternative Fixes

- **Launch Android Studio from a terminal** so it inherits your shell PATH:
  ```bash
  open -a "Android Studio"
  ```
  Fragile — breaks the next time someone double-clicks the app icon.

- **Build from the command line only**:
  ```bash
  npx expo run:android
  # or
  cd android && ./gradlew assembleDebug
  ```

- **Hard-code Node path in `android/settings.gradle`** — not recommended; non-portable across machines and CI.

### Why Not Edit `settings.gradle`?
The `commandLine("node", ...)` call is generated by Expo/React Native templates. Hard-coding `/opt/homebrew/bin/node` breaks Linux/Windows contributors and CI. Fixing the environment (symlink) is the portable solution.

## Node Version Pinning

This repo pins Node via two complementary mechanisms:

- **`.nvmrc`** — `22.21.0`. Run `nvm use` in the project root to switch.
- **`package.json` `engines`** — `node >=20.19 <23`, `npm >=10`. npm warns on mismatch; CI can enforce with `npm install --engine-strict`.

If you use `nvm`, add this to `~/.zshrc` to auto-switch on `cd`:

```bash
autoload -U add-zsh-hook
load-nvmrc() { [[ -f .nvmrc ]] && nvm use --silent; }
add-zsh-hook chpwd load-nvmrc
load-nvmrc
```

## Quick Health Check

```bash
node -v              # should match .nvmrc (22.21.0)
which node           # confirm location
ls -la /usr/local/bin/node   # confirm GUI-visible symlink exists
cd android && ./gradlew --version   # confirm Gradle can run
```
