Overview
Picup Media builds GemHub, a React Native app used by jewelry retailers and studios to capture product photos and videos, organize them by SKU (stockkeeping unit), and manage product listings for e-commerce and catalogs.
This exercise asks you to vibe-code a simplified, self-contained version of the core capture-and-catalog experience. We are not giving you access to our production codebase or backend. You should stand up your own lightweight data layer (local persistence, Supabase, Firebase, SQLite + file storage, etc.) — whatever lets you ship a credible demo quickly.
We care less about pixel-perfect branding and more about whether you can:
Ship a working RN app on iOS and Android
Implement the core product flows end-to-end
Design your own UX where it makes sense — you do not need to copy
GemHubʼs screens, tab order, or visual design
Use AI coding tools effectively while applying engineering judgment
What you are building
Think of it as “GemHub Lite” — a mobile app for a jewelry merchant who photographs inventory and needs every shot tied to a product SKU.
Required scope (must demonstrate)
Build a React Native app that exposes the areas below. Bottom tabs are a suggestion, not a requirement — use tabs, a drawer, a hub screen, or any navigation pattern you can justify if it makes the flows clearer.
Minimum three areas (names and layout are up to you):
Area	Purpose
Camera	Capture product photos (see below)
Media	Browse all captured/imported media
Products	Browse products organized by SKU
Plus Product detail (stack screen from Products or Media).
1. Camera
Full-screen (or primary) camera experience to take still photos of products.
After capture, show a preview / save step where the user must confirm a SKU before save:
	 	Type a SKU, pick an existing SKU, and/or offer “Generate SKU” that
creates a unique value (e.g. SKU-20260526-001 ) — your format choice, documented in README
SKU is always required; there is no “save without SKU” path
Optionally set a few basic fields (your choice — see Product sense below)
On save, media is always linked to that SKU — create the product if the SKU is new, or append media if it already exists
Photo capture only is sufficient. Video is a nice-to-have, not required.
2. Products
 	List or grid of products, each identified primarily by SKU.
Show at least: SKU, title (or placeholder), and a thumbnail (cover image).
Tapping a product opens Product detail.
3. Media
Gallery-style view of all media items (every item belongs to a product SKU .
Show enough context to identify the item (thumbnail + SKU and/or title).
Tapping media → open Product detail for that SKU.
Optional: adjustable grid density or list/grid toggle — bonus only (see below).
4. Product detail
View and edit core product info, e.g.:
SKU (may be read-only after create — your call)
Title
Description (short text)
Product type (simple picker or free text — e.g. Ring, Necklace, Earring) Media section: show all images for this product.
Cover/thumbnail for list views: default to the first image is enough. Letting the user pick a different cover is optional (bonus).
 	Ability to add more photos from the camera flow while on this product (navigate to camera with SKU pre-filled, return with new media attached).
You do not need catalogs/collections, pricing/inventory, gemstone appraisal, share links, analytics, or org/multi-tenant features.

Suggested user journeys (acceptance checklist)
Use these to self-test before submit:
New product from camera
Camera → capture photo → enter new SKU (typed or generated) + optional title → save → product appears under Products; photo appears under Media.
Add media to existing product
Open product detail → add photo → capture → same SKU pre-filled or selected → save → return to detail with new image in the list.
Existing SKU from camera
Camera → capture → choose existing SKU → save → media appended to that product (no duplicate product row).
Browse
Products and Media tabs stay in sync after saves; product list shows a thumbnail (first image is fine).
